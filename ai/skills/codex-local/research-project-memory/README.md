# Research Project Memory

面向长期研究项目的项目记忆 skill。它帮助 Codex 建立、读取、更新和演化项目记忆层，让未来的新对话可以从结构化上下文接上，而不是每次重新扫描整个仓库。

当前设计把项目记忆视为一个“嵌入式认知仓库”：`99-项目记忆/` 一开始只是项目里的普通文件夹，但目录结构从第一天起就按可迁移的方式组织，未来可以独立成 git 仓库、submodule、subtree，或者外部 memory repo。

## 适用场景

当一个项目需要的不只是普通笔记时，使用这个 skill：

- 长期研究项目、论文、毕业设计、实验、数据集、文献阅读；
- 希望未来 Codex 对话快速恢复上下文；
- 项目存在多条技术路线、研究假设、实验方向或写作主线；
- 需要把代码分支和思路分支分开记录；
- 项目交接、每日启动、每日收尾、记忆刷新、清理过期笔记。

核心原则：**代码分支表示实现状态，记忆分支表示认知状态**。两者可以互相引用，但不应该强行一一对应。

## 触发关键词

项目记忆类表达会触发这个 skill，例如：

- `建设项目记忆`
- `项目地图`
- `Codex 工作记忆`
- `快速熟悉项目`
- `不要全仓库扫描`
- `新对话接上`
- `同步一下`
- `收尾`
- `更新记忆`

认知分支类表达也会触发，例如：

- `我要尝试一个新方向`
- `开一个新思路`
- `分叉一下这个路线`
- `回到 XX 思路`
- `切回之前那个方向`
- `这个路线先挂起`
- `把 A 和 B 合流`
- `打一个里程碑`
- `记忆独立出来`
- `用 git 管理项目记忆`

## 标准目录结构

默认在项目内使用下面的结构：

```text
99-项目记忆/
  README.md
  manifest.md
  项目地图.md
  Codex工作记忆.md
  项目基础盘点.md
  Git仓库洞察.md
  项目内工作流.md

  memory/
    index.md
    branches/
      main.md
      route-*.md
    milestones/
    decisions/
    hypotheses/
    merges/
    references/

  logs/
    daily.md
    next.md
    discussions.md

  materials/

98-材料/
0-记录/
0-随记.md
0-讨论.md
```

已有项目可以保留旧路径。skill 应该通过链接和映射把旧路径纳入这套结构，而不是未经确认就移动历史笔记。

## 认知分支

`memory/branches/` 用来保存思考路线。一个路线可以对应一种方法、假设、实验方向、论文论证、产品策略或实现方案。

每条认知分支记录：

- 为什么出现这条路线；
- 核心假设是什么；
- 已观察到哪些证据；
- 关联了哪些代码分支、commit、实验、论文或讨论；
- 当前判断是什么；
- 下一步做什么；
- 这条路线的变更历史。

代码引用只是证据和实现上下文，不是认知分支的身份。一条思路可以跨多个代码分支；一个代码分支也可能只是某条思路的一部分实现尝试。

## 常见工作流

### 建设项目记忆

打开陌生项目或接手已有项目时，Codex 应该：

1. 用 `rg --files` 做有边界的文件盘点；
2. 优先检查 README、docs、配置、主入口、实验脚本、研究笔记等高信号文件；
3. 默认跳过大型生成物、缓存、模型输出和低信号二进制材料；
4. 创建或更新 `99-项目记忆/`；
5. 写入项目地图、Codex 工作记忆、项目基础盘点、Git 仓库洞察、memory manifest、认知分支索引和下一步。

### 尝试新方向

当用户说 `我要尝试一个新方向` 之类的话时，Codex 应该：

1. 读取 `memory/index.md` 和当前活跃路线；
2. 在 `memory/branches/` 下创建新的路线文件；
3. 记录这条路线为什么分叉、想验证什么；
4. 添加相关代码、实验、论文或讨论引用；
5. 更新 `memory/index.md` 中的活跃路线；
6. 如果当前工作方向改变，同步更新下一步。

### 回到旧思路

当用户说 `回到 XX 思路` 时，Codex 应该找到对应路线，将其设为活跃路线，记录为什么恢复它，并更新当前第一步行动。

### 暂停或归档路线

当某条路线被暂停或归档时，Codex 应该保存：

- 为什么暂停；
- 什么证据会让它值得恢复；
- 哪些工作不要重复做。

### 思路合流

当两条路线收敛时，Codex 应该在 `memory/merges/` 下创建合流记录，并更新源路线和目标路线：哪些判断被吸收，哪些被放弃，哪些问题仍需观察。

### 创建里程碑

当一个阶段形成稳定结论时，Codex 应该在 `memory/milestones/` 下创建日期化里程碑，并从活跃路线和 `memory/index.md` 链接过去。

## Git 管理方式

项目记忆默认先嵌入在项目内。复杂度上来后，可以提升为：

- `embedded`：继续作为宿主项目里的普通文件；
- `standalone`：在 `99-项目记忆/` 内初始化独立 git 仓库；
- `submodule`：记忆放在独立仓库，并作为宿主项目的 submodule；
- `external`：记忆放在外部位置，反向引用一个或多个宿主项目。

skill 在迁移前应该先检查并补齐结构。除非用户明确要求，不应自动初始化、移动、推送、合并或改写 git 状态。

## 文件说明

- `SKILL.md`：Codex 实际执行的 skill 指令。
- `README.md`：给人看的概览说明，也方便未来维护这个 skill。

## 设计原则

项目记忆应该帮助未来思考，而不是制造第二套繁琐流程。根入口保持短，长材料放到支撑文件；不确定性要明说；每个长期判断都应该能追溯到证据。
